# SubTrans.ai - 영어 자막 한글 번역 서비스 개발 세션 노트 (Session Notes)

이 문서는 영어 자막(.srt) 파일을 업로드/스캔하여 [전처리] -> [Gemini 번역 및 검증] -> [후처리] 과정을 거쳐 한글 자막을 완벽한 싱크로 생성하는 로컬 웹 서비스 개발에 대한 전체 히스토리와 설계 요약을 제공합니다.

---

## 1. 프로젝트 개요 및 요구사항
- **목적**: 로컬 환경에서 영어 자막 파일(.srt)을 입력받아 번역 전처리, Gemini AI 모델 번역, 정합성 검증, 번역 후처리를 자동 수행하여 최종적으로 원본 자막과 동일한 타임라인 싱크를 갖는 `a (kr).srt` 자막을 생성하는 서비스 구축.
- **주요 UI 요구사항**: 
  - 로컬 디렉토리 경로를 지정하여 일괄 스캔하는 대시보드 화면 제공.
  - 실시간으로 진행률바와 상세 작업 로그(터미널 스타일)를 표출하고 작업 완료 시 브라우저 알림 기능 제공.
  - 웹 화면 및 웹 서버 백엔드 콘솔(표준 출력) 모두에 진행상황 출력.

---

## 2. 세부 파이프라인 흐름 및 설계

### [1단계: 전처리]
- 번역 시작 전 `pre_srt.py` 라이브러리를 임포트하여 실행.
- 영어 문장 구분 조정 및 `filters.json`에 정의된 노이즈 텍스트(예: `[applause]`, `[laughter]`)를 제거하고 `a_pre.srt` 임시 자막 생성.

### [2단계: 100개 단위 청크 번역 & 검증 재시도]
- `a_pre.srt` 파일을 로드하여 **100개 자막 블록** 단위의 청크로 분할하여 번역.
- **번역 지침**:
  1. 원본의 라인 인덱스와 타임스탬프를 1:1로 엄격하게 일치시킴 (단 1밀리초의 오차도 허용 안 함).
  2. 반말(해라체)을 배제하고 자연스럽고 간결한 존댓말 회화체('요'/'습니다') 사용.
  3. 번역 결과물 이외의 부가 설명이나 마크다운 백틱(` ``` `) 등은 파싱 단계에서 필터링.
- **오류 감지 및 자동 재시도 (Retry Loop)**:
  - **429 Resource Exhausted (할당량 초과)**: 감지 시 API에서 요구하는 시간(또는 기본 20초)만큼 대기한 후 최대 3회 재시도.
  - **503 Unavailable (서버 부하)**: 감지 시 8초 대기 후 최대 3회 재시도.
  - **1차 번역 검증 실패**: 개수나 타임스탬프 불일치 시, 즉시 **대체(Fallback) 모델**을 사용해 재시도. 대체 모델도 3회 재시도 실패 시 작업을 중단하고 에러 리포트 표출.

### [3단계: 최종 정합성 검증]
- 청크별 번역이 모두 완료되면 하나의 파일로 병합하여 `a (kr).srt`로 임시 저장.
- 새롭게 작성한 `verify_srt.py`를 실행하여 원본 전처리 파일과 번역 완성 파일의 전체 자막 개수, 모든 인덱스 번호, 시작/종료 타임스탬프가 1:1로 정확하게 일치하는지 최종 검증.

### [4단계: 후처리]
- 정합성 검증이 통과되면 `post_srt.py`를 실행하여 한국어 자막에 맞지 않는 불필요한 추임새(예: "어, ", "음, ", ", 여러분.")를 정규식으로 정리.
- 후처리를 거친 최종 결과물을 `a (kr).srt`로 덮어쓰고, 후처리 과정에서 생성된 다른 임시 파일들(예: `fixed_*.srt`)은 자동으로 삭제하여 디렉토리를 깨끗하게 정리.

## 3. 파일 레이아웃 및 변경점

```
media_utils/
│
├── static/                   # 프론트엔드 정적 파일
│   ├── index.html            # 글래스모피즘 기반의 번역 관리 대시보드
│   ├── style.css             # HSL 컬러, 다크 모드, 애니메이션 스타일시트
│   └── app.js                # 알림 권한 획득, SSE 실시간 로그 파싱 및 뷰어
│
├── .env                      # Gemini API Key 설정 파일 (GEMINI_API_KEY)
├── requirements.txt          # fastapi, uvicorn, google-genai, python-dotenv 추가
├── web_server.py             # FastAPI 백엔드 웹 서버 (SSE 스트리밍, uvicorn 실행)
├── verify_srt.py             # 자막 1:1 타임스탬프/개수 정합성 검증 전용 스크립트
├── post_srt.py               # (수정) CP949 인코딩 크래시 방지를 위해 이모지를 ASCII로 변경
├── pre_srt.py                # 번역 전처리용 기존 스크립트
└── filters.json              # 전처리용 노이즈 필터 정의
```

### 파일별 주요 수정/구현 코드 상세

#### 1. `verify_srt.py` (자막 정합성 검증 스크립트)
두 자막의 타임라인 일치율을 엄격하게 대조하기 위해 직접 생성한 코드입니다.
```python
# verify_srt.py 핵심 검증 함수
def verify_srt_files(pre_file_path, kr_file_path):
    # (코드 중략) 두 파일을 파싱하여 개수 대조 후 루프 작동
    for i in range(pre_len):
        if pre_b['index'] != kr_b['index']:
            return False, "인덱스 불일치"
        if pre_b['start'] != kr_b['start'] or pre_b['end'] != kr_b['end']:
            return False, "타임스탬프 불일치"
    return True, "모든 싱크 일치 확인"
```

#### 2. `web_server.py` (API 서버 & SSE 비동기 번역 루프)
- FastAPI로 백엔드를 구현하고 정적 파일을 마운트(`app.mount("/static", ...)`)하여 프론트엔드를 서빙합니다.
- `requests.POST /api/translate` 요청 시 번역 과정을 Server-Sent Events(`text/event-stream`) 형태로 전송하며, 터미널 표준 출력(`print(..., flush=True)`)을 동시에 수행합니다.
- API 호출 시 `google-genai` 라이브러리를 이용하여 비동기 스레드 풀(`run_in_executor`) 내에서 처리합니다.

#### 3. `post_srt.py` (출력 인코딩 크래시 수정)
윈도우 환경(CP949 로케일)에서 파이썬 실행 시 터미널 표준 출력의 ✅(`\u2705`) 및 ❌(`\u274c`) 이모지 문자를 인코딩하지 못하고 프로세스가 크래시되는 현상이 발견되어, 문구를 ASCII 텍스트(`[OK]`, `[FAIL]`)로 대체 변경하였습니다.

## 4. API 모델 매칭 팁
사용자 환경 및 구글 API Studio 무료 등급 기준, 사용 가능한 모델 ID 리스트는 다음과 같습니다.
- **기본 모델 (Primary)**: `gemini-3.5-flash` (Frontier 등급의 초고속 속도 및 넉넉한 쿼터 제공)
- **대체 모델 (Fallback)**: `gemini-3.1-pro-preview` (일반적인 `gemini-3.1-pro` ID는 API v1beta 환경에서 지원되지 않으므로, 반드시 뒤에 `-preview`가 붙은 공식 ID를 호출해야 함)

## 5. 실행 및 관리 가이드

### API Key 설정 (`.env`)
프로젝트 폴더 내 `.env` 파일에 아래와 같이 유효한 API 키를 저장합니다.
```env
GEMINI_API_KEY=AIzaSy...
```

### 웹 서버 구동
가상 환경(`.venv`)의 파이썬을 이용해 웹 서버를 실행합니다.
```powershell
.venv\Scripts\python web_server.py
```
서버 구동 후 브라우저에서 **[http://127.0.0.1:8000](http://127.0.0.1:8000)** 에 접속하여 디렉토리 경로 입력 후 번역을 실행하면 됩니다.
